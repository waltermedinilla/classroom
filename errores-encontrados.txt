1. XSS MASIVO VIA innerHTML (60+ lugares)
   Archivo: public/js/course.js
   Líneas: 268-293, 340-351, 546-554, 711-725, 788-801, 1162-1164, 1269,
           1332-1334, 1603-1615, 1805-1822
   Problema: Se interpola contenido de usuarios (anuncios, comentarios,
   actividades, nombres de archivo, etc.) directamente en innerHTML sin
   sanitización. Cualquier alumno puede ejecutar JavaScript arbitrario
   en el navegador de todos los docentes y compañeros.

2. XSS EN ATTACHMENTS - escAtt() incompleto
   Archivo: public/js/course.js
   Línea: 35, 47-79, 114
   Problema: La función escAtt() solo escapa & y ", pero no <, > ni '.
   Las URLs de archivos se inyectan sin sanitizar en atributos src/href
   de iframes y links de descarga, permitiendo inyección de atributos.

3. CSRF - SIN PROTECCIÓN EN NINGÚN ENDPOINT
   Archivo: TODOS (routes/*.js, public/js/*.js, views/*.ejs)
   Problema: Ninguna de las rutas POST/PUT/DELETE implementa tokens
   CSRF ni SameSite=Strict/Lax. Un atacante puede engañar a un usuario
   autenticado para crear cursos, publicar anuncios, cambiar notas,
   eliminar usuarios, etc. desde un sitio malicioso.

4. REGISTRO SIN VALIDACIÓN DE ROL - ESCALADA A ADMIN
   Archivo: routes/auth.js
   Línea: 37-38
   Código:
     const { name, email, password, role } = req.body;
     const userRole = role || 'student';
     const user = await User.create({ name, email, password, role: userRole });
   Problema: No hay whitelist server-side de roles permitidos. Un
   atacante puede enviar role: "admin" en el POST /register y
   registrarse como administrador.
   Contraste: la ruta invite (línea 114-115) sí tiene whitelist:
     const allowed = ['student', 'teacher', 'preceptor', 'soe', 'directivo'];
     const userRole = allowed.includes(role) ? role : 'student';

5. PATH TRAVERSAL EN SUBIDA DE ARCHIVOS
   Archivo: routes/announcements.js
   Línea: 20
   Código:
     const courseId = req.body.courseId || 'general';
     const dir = path.join(ARCHIVOS_BASE, schoolId, 'novedades', courseId);
   Problema: req.body.courseId se usa directamente en la construcción
   de rutas de archivos sin sanitizar. Un atacante puede enviar
   courseId = "../../../some/directory" para escribir archivos en
   directorios arbitrarios. El middleware multer se ejecuta ANTES de
   la verificación de autorización (línea 103-107).

6. PATH TRAVERSAL EN ACTIVIDADES
   Archivo: routes/activities.js
   Línea: 39
   Problema: Mismo patrón que #5. req.body.courseId usado directamente
   en path.join() para almacenar archivos. Multer corre antes que la
   autorización (línea 150).

7. DUPLICATE ENROLLMENT - includes() no funciona con ObjectId
   Archivo: routes/courses.js
   Línea: 177
   Código:
     if (course.students.includes(req.userId)) {
       return res.status(400).json({ error: 'Ya estás en este curso' });
     }
   Problema: course.students es un array de ObjectId, req.userId es
   un string. Array.includes() usa === que nunca matchea ObjectId
   con string. Un estudiante puede unirse al mismo curso infinitas
   veces. En otras partes del código (líneas 259, 293) usan
   .toString() correctamente.

10. CÓDIGO DE CURSO COLISIONA (birthday paradox)
    Archivo: models/Course.js
    Línea: 29
    Código:
      code: { type: String, default: () => uuidv4().slice(0, 6).toUpperCase() }
    Problema: UUID v4 en hex = 16^6 = 16.7M combinaciones. Con ~4,800
    cursos hay ~50% de colisión. Con 10,000 hay >95%. El índice unique
    causará fallos aleatorios al guardar.

11. NO HAY CASCADE DELETE AL BORRAR USUARIO
    Archivo: routes/admin.js
    Líneas: 227-244
    Problema: Al borrar un usuario no se:
    - Elimina de course.students
    - Reasigna course.owner
    - Borra sus submissions
    - Elimina sus notas en activities
    - Limpia avatar del disco
    Si se borra un docente, sus cursos quedan huérfanos.

12. NO HAY CASCADE DELETE AL BORRAR ESCUELA
    Archivo: routes/superadmin.js
    Líneas: 146-153
    Problema: Al borrar una escuela no se borran ni reasignan sus
    usuarios, cursos, materias ni tokens de invitación. Todo queda
    huérfano.

13. SIN RATE LIMITING EN NINGÚN ENDPOINT
    Archivo: server.js y todas las rutas
    Problema: Los rate limiters declarados en server.js (generalLimiter,
    authLimiter, uploadLimiter) están configurados pero el login/register
    no tienen protección contra fuerza bruta más allá del authLimiter
    (15 intentos/15min). El resto de endpoints no tienen límites.

15. SERVER EMPIEZA ANTES QUE LA DB
